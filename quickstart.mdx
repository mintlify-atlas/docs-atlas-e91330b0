## Run your first workflow

<Steps>
  <Step title="Install act">
    Follow the [installation guide](/installation) to install act and confirm Docker is running.

    ```bash
    act --version
    ```
  </Step>
  <Step title="Navigate to a repository with workflows">
    `act` looks for workflow files in `.github/workflows/` relative to your current directory.

    ```bash
    cd /path/to/your/repo
    ls .github/workflows/
    ```

    If you don't have a repository with workflows yet, create a minimal one:

    ```bash
    mkdir -p .github/workflows
    cat > .github/workflows/hello.yml << 'EOF'
    name: Hello
    on: push
    jobs:
      greet:
        runs-on: ubuntu-latest
        steps:
          - run: echo "Hello from act!"
    EOF
    ```
  </Step>
  <Step title="Run act">
    With no arguments, `act` triggers the `push` event and runs all jobs that respond to it.

    ```bash
    act
    ```

    If your workflows only define one event, act uses that event automatically instead of defaulting to `push`.
  </Step>
  <Step title="Choose a Docker image size">
    On the first run, act asks which Docker image size to use as the default runner environment. This setting is saved to your `.actrc` config file and used for future runs.

    ```
    Please choose the default image you want to use with act:
      - Large size image: ca. 17GB download + 53.1GB storage...
      - Medium size image: ~500MB, includes only necessary tools...
      - Micro size image: <200MB, contains only NodeJS...

    Default: Medium
    ```

    | Size | Approximate download | Contents |
    |---|---|---|
    | Large | ~17 GB | Snapshots of GitHub-hosted runners (`catthehacker/ubuntu:full-*`). Most compatible. |
    | Medium | ~500 MB | Tools needed to bootstrap actions (`catthehacker/ubuntu:act-*`). Compatible with most actions. |
    | Micro | &lt;200 MB | Node.js only (`node:16-buster-slim`). Fastest, but many actions will not work. |

    <Tip>
      Medium is the best starting point for most workflows. It downloads quickly and supports the majority of common actions. Switch to Large only if you encounter compatibility issues that Medium cannot resolve.
    </Tip>
  </Step>
  <Step title="Watch the workflow execute">
    act streams step output to your terminal in real time, with color-coded job names. A successful run ends with a summary similar to:

    ```
    [Hello/greet] 🐳 docker pull image=node:16-buster-slim ...
    [Hello/greet]   ✅  Success - Main echo "Hello from act!"
    [Hello/greet] 🏁  Job succeeded
    ```
  </Step>
</Steps>

## Common commands

**List all workflows and jobs**

```bash
act --list
```

This prints a table of every job defined in your workflows along with the event that triggers it and the runner label it requires.

**Run a specific job**

```bash
act -j <job-id>
```

Replace `<job-id>` with the job identifier from `act --list`. This runs only that job, regardless of which event triggers it.

**Trigger a specific event**

```bash
# Trigger the push event (default)
act push

# Trigger a pull_request event
act pull_request
```

Pass the event name as the first argument. `act` runs all jobs that are triggered by that event.

## Persisting settings with .actrc

act reads configuration from `.actrc` files in the following order:

1. XDG config directory (e.g., `~/.config/act/actrc` on Linux)
2. `~/.actrc` in your home directory
3. `.actrc` in the directory where you run `act`

Each line in an `.actrc` file is treated as a CLI flag. For example, to set a default platform mapping without typing it every time:

```
-P ubuntu-latest=catthehacker/ubuntu:act-latest
-P ubuntu-22.04=catthehacker/ubuntu:act-22.04
-P ubuntu-20.04=catthehacker/ubuntu:act-20.04
```

<Note>
  The first-run image survey writes your selection to the XDG config directory path automatically. You can edit the file at any time to change the default image or add other persistent flags.
</Note>
