---
title: "Secrets and Variables"
description: "How to supply secrets, variables, environment variables, and workflow dispatch inputs to act when running GitHub Actions locally."
---

Act supports four distinct data types that map to the corresponding GitHub Actions contexts: secrets (`secrets.*`), variables (`vars.*`), environment variables (`env.*`), and workflow dispatch inputs (`inputs.*`). Each can be supplied via CLI flags or files.

## Secrets

Secrets are available inside workflows via the `secrets` context (e.g., `${{ secrets.MY_SECRET }}`).

### CLI flag

Pass a secret inline:

```bash
act -s MY_SECRET=supersecretvalue
```

Omit the value to have act prompt you interactively:

```bash
act -s MY_SECRET
```

Pass multiple secrets:

```bash
act -s DB_PASSWORD=abc123 -s API_KEY=xyz789
```

### Secret file

Store secrets in a dotenv-formatted file and reference it with `--secret-file`. The default path is `.secrets` in the working directory:

```bash
# use the default .secrets file
act

# use a custom file
act --secret-file .env.secrets
```

Example `.secrets` file:

```bash
# .secrets
MY_SECRET=supersecretvalue
DB_PASSWORD=abc123
API_KEY=xyz789
GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx
```

<Note>
Secret keys are normalized to uppercase when loaded from a file (`readEnvsEx` is called with `caseInsensitive: true`), so `my_secret` and `MY_SECRET` are treated as the same key.
</Note>

### GITHUB_TOKEN

Act automatically populates `GITHUB_TOKEN` from the `gh` CLI if it is installed and authenticated. If `GITHUB_TOKEN` is already present in your secrets (via flag or file), that value is used instead:

```go
if _, hasGitHubToken := secrets["GITHUB_TOKEN"]; !hasGitHubToken {
    secrets["GITHUB_TOKEN"], _ = gh.GetToken(ctx, "")
}
```

You can also pass it explicitly:

```bash
act -s GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx
```

### Secret masking

<Warning>
By default, secret values are masked in all log output — they appear as `***`. Do **not** use `--insecure-secrets` in production or shared environments, as it disables masking and prints secret values in plain text:

```bash
# NOT RECOMMENDED — only for local debugging
act --insecure-secrets
```
</Warning>

## Variables

Variables are available in workflows via the `vars` context (e.g., `${{ vars.MY_VAR }}`). Unlike secrets, variables are not masked in logs.

### CLI flag

```bash
act --var MY_VAR=hello
act --var ENV_NAME=production --var REGION=us-east-1
```

### Variable file

The default file path is `.vars`:

```bash
# use the default .vars file
act

# use a custom file
act --var-file config/vars.env
```

Example `.vars` file:

```bash
# .vars
ENV_NAME=production
REGION=us-east-1
FEATURE_FLAG=true
```

## Environment variables

Environment variables are injected directly into job containers (accessible as `$MY_ENV` in shell steps). They are distinct from the `vars` context.

### CLI flag

```bash
act --env MY_ENV=value
act --env NODE_ENV=test --env LOG_LEVEL=debug
```

### Environment file

The default file path is `.env`:

```bash
# use the default .env file
act

# use a custom file
act --env-file .env.local
```

Example `.env` file:

```bash
# .env
NODE_ENV=test
LOG_LEVEL=debug
DATABASE_URL=postgres://localhost:5432/mydb
```

<Tip>
`.yml` and `.yaml` files are also supported for env, var, and secret files. Act detects the extension and parses them accordingly.
</Tip>

## Action inputs (workflow_dispatch)

Workflow dispatch inputs are available in workflows via the `inputs` context (e.g., `${{ inputs.environment }}`). These correspond to the `inputs:` defined under `on.workflow_dispatch`.

### CLI flag

```bash
act workflow_dispatch --input environment=staging
act workflow_dispatch --input environment=production --input dry-run=false
```

### Input file

The default file path is `.input`:

```bash
# use the default .input file
act workflow_dispatch

# use a custom file
act workflow_dispatch --input-file config/inputs.env
```

Example `.input` file:

```bash
# .input
environment=staging
dry-run=false
notify=true
```

<Note>
When inputs are supplied via `--input` or `--input-file`, act constructs the event JSON automatically as `{"inputs": {...}}`. You do not need a separate `-e` event file for `workflow_dispatch` inputs.
</Note>

## File precedence

For each data type, CLI flags take priority over file values. Values already set by a flag are not overwritten by the corresponding file:

```go
// from cmd/root.go — CLI flag values are parsed first, then the file fills in missing keys
envs := parseEnvs(input.envs)
_ = readEnvs(input.Envfile(), envs)
```

This means you can keep a base `.secrets` file and override individual values on the command line without modifying the file.
